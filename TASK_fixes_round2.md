## Задача 1 (главная). Маршрутизация эффектов эмоций на рёбра

**Проблема.** Сейчас эффекты всех эмоций события применяются к одному
контрагенту с fallback-логикой: если агент события — сам воспринимающий,
эффект перенаправляется на пациента. В результате в тестовом сценарии
персонаж 1, причинивший вред персонажу 2 и испытавший distress, ПОНИЗИЛ
свой liking к пострадавшему (liking 1→2 = −0.29) — «обвинение жертвы»,
прямо противоречащее статье (ср. сцену допроса: distress грабителя от
воспоминания о похитителях НЕ меняет liking к полицейскому, потому что
«the policeman is not responsible»).

**Принятая модель маршрутизации (строго по формальным правилам статьи):**

1. Каждая порождённая эмоция несёт явную мишень социального эффекта —
   `causeId` = сущность, ВЫЗВАВШАЯ эмоцию. Для эмоций от события
   `causeId = evt.AgentId`. Никогда не пациент.
2. Каналы (все адресуются causeId):
   - liking (Fig. 4): позитивная эмоция, вызванная j → liking к j растёт;
     негативная → падает.
   - dominance (Fig. 5): pride/anger, вызванные j → dominance над j растёт;
     fear/distress/admiration/shame, вызванные j → падает.
   - solidarity (Fig. 6): негативная эмоция, вызванная j → solidarity к j падает;
     позитивные эмоции solidarity НЕ меняют (рост — только через конгруэнтность
     выраженных эмоций в `ObserveExpression`, этот путь не трогать).
3. **Если causeId == perceiverId (самопричинённая эмоция) — социальный эффект
   отбрасывается целиком.** Рёбер «к себе» не существует. Никакого fallback
   на пациента или другого участника события.
4. Эмоция при этом остаётся в эмоциональном состоянии NPC как обычно —
   отбрасывается только её вклад в социальное отношение.

**Известное ограничение (НЕ решать в этом раунде, только задокументировать):**
пример IV-B статьи (падение dominance грабителя в диаде с полицейским при
причине-похитителях) формальным правилам не соответствует и по тексту статьи
неразрешим — у реплики нет формального события. Решение отложено до введения
понятия «разговорный контекст» (отдельный будущий этап). В `CLAUDE.md` — в
раздел открытых вопросов, с пометкой «маршрутизация по causeId выбрана как
буквальное прочтение Fig. 4–6».

**Приёмка** — эталонный сценарий (персонажи/оценки/словарь как в ручном тесте
от 2026-07-07, воспроизвести как интеграционный тест):
- Персонажи: 1(e=0.60,n=0.54), 2(e=0.10,n=−0.23), 3(e=0.32,n=−0.66);
  attitudes: 1→1:0.90, 1→2:0.20, 2→1:−0.10, 2→2:0.22, 3→3:0.67, 3→1:0.80;
  действия: 1(effect 0.50), 2(effect −0.56);
  praise: owner1/action1=1.00, owner2/action2=0.50, owner3/action0=0.00.
- События (dc=1, t=0): ⟨p=1,a=1,act=2,pat=2⟩, ⟨p=2,a=1,act=1,pat=1⟩,
  ⟨p=3,a=2,act=1,pat=1⟩, ⟨p=3,a=1,act=2,pat=2⟩.
- Ожидаемые эмоции (не должны измениться, кроме исчезновения Admiration —
  см. задачу 2): 1: Distress≈0.48; 2: Distress≈0.27; 3: Joy≈0.75.
- Ожидаемые отношения:
  - 1→2: liking=0.00, dominance=0.00 (самопричинённый distress отброшен;
    было −0.29/−0.29 — это и есть исправляемый баг);
  - 2→1: liking≈−0.12 (distress вызван агентом 1 — канал работает как раньше),
    dominance≈−0.12, solidarity=0.00 (падение от негативной эмоции с клампом
    снизу на 0 — проверить, что дельта отрицательная до клампа);
  - 3→2: liking≈+0.34 (joy вызвана агентом 2), dominance=0.00 (admiration
    исчезает после задачи 2);
  - 3→1: все нули (fortunes-of-others в модели Ochs исключены — это НЕ баг);
  - familiarity=0.00 везде (инвариант раунда 1).

## Задача 2. Утечка субъективного praise между владельцами

**Проблема.** В сценарии выше у персонажа 3 вспыхнула Admiration=0.20, хотя
praise₃(action 1) в таблице не задан (у владельца 3 есть только запись для
action 0). Неопределённый praise обязан давать 0 → ветка admiration не должна
срабатывать. Значение 0.20 указывает, что lookup praise при промахе берёт
чужую запись (другого владельца) или ненулевой дефолт.

**Что сделать.** Найти lookup субъективного praise (вероятно, `Simulation` или
`ActionDictionary`); при отсутствии записи ⟨owner, action⟩ возвращать строго 0.
Никаких fallback на глобальное значение или записи других владельцев.

**Приёмка.** Юнит-тест: praise запрошен для ⟨owner, action⟩ без записи → 0;
запись другого владельца для того же action на результат не влияет.
Интеграционно: в эталонном сценарии Admiration₃ отсутствует, dominance 3→2 = 0.

## Задача 3. Единый источник правды для attitude «к себе»

**Проблема.** Attitude к себе теперь живёт в двух местах: колонка «Attitude к
себе [-1..1]» на вкладке «Персонажи» и строки owner==target на вкладке
«Субъективные оценки». Сейчас значения совпадают, но два хранилища могут
разойтись, и неясно, какое читает аппрейзал.

**Что сделать.** Одно хранилище — общий словарь attitudes (owner → target →
значение), self-attitude = запись owner==target. Колонка на вкладке «Персонажи»
либо удаляется, либо становится представлением той же записи (двусторонняя
привязка к единственному хранилищу). Запретить дублирующие строки
⟨owner, target⟩ в таблице оценок (при попытке добавить дубль — редактировать
существующую).

**Приёмка.** Изменение значения в любом из двух мест UI видно во втором;
аппрейзал использует единственное хранилище (юнит-тест на `NpcState`);
добавить дубль ⟨owner, target⟩ невозможно.
