# Задание: исправления по итогам ручного тестирования UI (раунд 1)

> Контекст: ручной прогон RelationshipSimulator выявил, что эмоции и отношения
> оставались нулевыми при любых сценариях. Диагноз подтверждён экспериментом:
> формулы ядра работают (Joy=0.98, liking=0.29 при событии с patientId=perceiverId),
> проблема — в непрокинутых входных данных (attitudes, praiseworthiness) и в одном
> семантическом решении (familiarity). Ниже 4 задачи. Задачи 1–3 механические,
> задача 4 — зафиксированное архитектурное решение, его не пересматривать без
> обсуждения с автором.

## Инварианты (нарушать нельзя ни в одной задаче)

- `Dynamics` / `Dynamics.Rules` не ссылаются на `Graphs` и не принимают `INode` —
  только `int entityId` и значения.
- `SocialRelation` — immutable sealed class. Никаких мутаций полей: обновление
  ребра — всегда присваивание нового объекта.
- Движок пишет в граф только через `AddDirectEdge(new Edge(from, to, updated))`;
  диспетчеризация в историю происходит внутри `DeepGraph.AddEdge` (HistoryEdge).
  Не менять этот путь.
- Поиск ребра — по тройке `(from, to, тип Relationship)` (WideEdge). Не сводить к паре.
- UI-логика не просачивается в `RelationshipCore`.
- После каждой задачи: `dotnet test` — все существующие тесты (110) зелёные,
  плюс новые тесты на само изменение.

## Задача 1. Attitudes к произвольным сущностям

**Проблема.** В UI (вкладка «Персонажи») редактируется только «Attitude к себе».
Attitude воспринимающего к пациенту события задать невозможно, поэтому аппрейзал
для любого события с patientId ≠ perceiverId получает att=0 → стимул нулевой →
эмоций нет → отношения не меняются. Это была первопричина «всё по нулям».

**Что сделать.**
1. Проверить, как `NpcState` хранит attitudes: если это скаляр «к себе» —
   заменить на словарь `entityId -> attitude` (диапазон [-1..1]), «к себе» становится
   частным случаем `attitudes[selfId]`. Если словарь уже есть — только UI.
2. UI: дать редактировать attitudes персонажа к любой существующей сущности
   (персонажи и объекты словаря). Форма — на усмотрение (доп. таблица на вкладке
   персонажа или отдельная вкладка «Attitudes»), главное — значения должны
   попадать в `NpcState` до прогона сценария.

**Приёмка.**
- Юнит-тест `Simulation`: событие ⟨agent=2, action(effect>0), patient=3⟩ при
  att₁(3)>0 вызывает у перцевера 1 эмоцию joy-семейства; при att₁(3)<0 — distress-семейства
  (сверка типов по Fig. 2 статьи Ochs).
- Ручная проверка: сценарий из скриншотов (perceiver=1, agent=2, action=1, patient=2)
  при заданном att₁(2)≠0 даёт ненулевые эмоции и сдвиг liking 1→2.

## Задача 2. Praiseworthiness действий

**Проблема.** В `ActionDictionary`/UI у действия есть только «Эффект». Интенсивность
ветки pride/shame/admiration/anger по статье считается от praiseworthiness действия —
без этого поля вся ветка мертва (admiration не вспыхивает даже при позитивном событии
с внешним агентом), а вместе с ней мёртв и основной канал изменения dominance.

**Что сделать.**
1. Проверить, есть ли praiseworthiness в типе действия в `Dynamics`. Если нет —
   добавить (диапазон [-1..1], 0 = нейтрально). Если есть — только UI.
2. UI: колонка «Praise [-1..1]» в словаре действий.
3. Убедиться, что `EmotionRules` использует praise при вычислении стимула для
   pride/shame/admiration/anger (по IV-A: avg с участием |praise| и dc).

**Приёмка.**
- Юнит-тест `Rules`: действие с praise>0, агент — другой персонаж → у перцевера
  admiration; praise<0 → anger. Агент = сам перцевер → pride / shame соответственно.
- Интеграционный тест `Simulation`: событие с praise<0 от агента 2 → anger у
  перцевера 1 → dominance 1→2 растёт; событие, вызывающее admiration → dominance падает
  (направления по Fig. 5 статьи).

## Задача 3. Валидация диапазонов на вводе в UI

**Проблема.** Поле dc подписано [0..1], но UI молча принял 5.00 (ядро заклампило).
Молчаливый кламп маскирует ошибки пользователя при постановке экспериментов.

**Что сделать.**
- Валидация всех числовых полей UI по их диапазонам: extraversion/neuroticism/
  attitude/effect/praise ∈ [-1..1], dc ∈ [0..1], время ≥ 0.
- Поведение при выходе за диапазон: не давать зафиксировать значение (подсветка
  ячейки + не пропускать в модель), НЕ молчаливый кламп.
- Ядро не трогать: существующие клампы в `RelationshipCore` остаются как вторая
  линия обороны (защита программного API).

**Приёмка.** Ввод 5.00 в dc не проходит в сценарий; визуальная индикация ошибки.
Существующие тесты ядра не изменяются.

## Задача 4. Семантика familiarity: заморозить эмоциональный дрейф (решение принято)

**Проблема.** Прогон показал familiarity=0.01 после одного эмоционального события.
Статья Ochs противоречит сама себе: раздел III-D говорит, что эмоции НЕ имеют
прямого влияния на familiarity (она растёт только через передачу конфиденциальной
информации — механизм вне статьи); раздел IV-A описывает f_sr как увеличивающую
«каждое измерение». Текущий код реализует IV-A буквально.

**Принятое решение (не пересматривать в рамках задачи): следуем III-D.**
Обоснование, кратко: familiarity по определению — мера информационного обмена,
а не аффекта; в эталонном сценарии статьи авторы не показывают её изменение от
эмоций, но используют как порог; §6.2.4 диссертации О'Коннора трактует familiarity
так же (убывает от отсутствия взаимодействия) — заморозка сейчас развязывает руки
будущему decay на базе HistoryEdge.

**Что сделать.**
1. В `SocialRelationRules` (функция дельты f_sr): вклад эмоций в familiarity = 0.
   Familiarity из `Perceive`/`ObserveExpression` не меняется никогда.
2. Оставить явную точку расширения: отдельный публичный метод в
   `SocialRelationRules` вида `FamiliarityFromInformationTransfer(current, confidentiality)`
   — пока может быть не подключён к движку, но должен существовать и быть покрыт
   юнит-тестом, чтобы будущий механизм передачи информации имел готовый вход.
3. Обновить интеграционные тесты, если какие-то из 9 эталонных сценариев
   ассертят ненулевую familiarity, полученную от эмоций.
4. Зафиксировать решение в `CLAUDE.md`: раздел «Осознанные трактовки статьи Ochs»,
   рядом с решением по counterparty. Формулировка: «familiarity следует III-D,
   а не буквальному IV-A; эмоции на неё не влияют; канал изменения — только
   явная передача информации (метод-задел) и будущий decay (§6.2.4 O'Connor)».

**Приёмка.**
- Юнит-тест: любой эмоциональный вектор на входе f_sr даёт нулевую дельту familiarity.
- Юнит-тест на метод-задел: рост familiarity от confidentiality монотонный, результат в [0..1].
- Сценарий из скриншотов после правки: familiarity=0.00, liking по-прежнему ~0.29.
